CLIProxyAPI 部署教程与使用说明

CLIProxyAPI 部署教程与使用说明：把 CLI AI 工具封装成统一 API 服务

适用对象：想把 Gemini CLI、Claude Code、ChatGPT Codex、Antigravity 等 CLI/OAuth 工具封装成 OpenAI、Gemini、Claude、Codex 兼容 API 的用户。
项目地址：https://github.com/router-for-me/CLIProxyAPI

一、CLIProxyAPI 是什么？

CLIProxyAPI 是一个代理服务，它可以把 CLI 类 AI 工具包装成标准 API 服务，对外提供 OpenAI、Gemini、Claude、Codex 等兼容接口。

简单来说，它的作用是：把原本只能通过命令行或 OAuth 登录使用的模型能力，转换成可以被 Cherry Studio、Open WebUI、OpenCode、Claude Code、SDK 或自写程序调用的 API。

它支持的核心能力包括：

  • 流式与非流式响应；
  • 函数调用；
  • 工具调用；
  • 多模态输入；
  • 多账号轮询负载均衡；
  • OAuth 登录；
  • OpenAI 兼容上游服务接入；
  • OpenAI、Gemini、Claude、Codex 等多种 API 格式兼容。

二、典型使用场景

CLIProxyAPI 适合以下几类场景：

1. 统一多个 AI CLI 工具的调用入口

例如把 Gemini CLI、Claude Code、Codex 等统一封装成：

1

http://localhost:8317/v1/chat/completions

这样一来，原本不同来源、不同协议、不同登录方式的模型，就可以通过统一接口进行调用。

2. 让普通 OpenAI API 客户端调用 CLI 模型

只要客户端支持自定义 Base URL 和 API Key，就可以接入 CLIProxyAPI。

常见客户端包括：

  • Cherry Studio；
  • Open WebUI；
  • OpenCode；
  • Continue；
  • 自写 Python / JavaScript 程序；
  • 其他支持 OpenAI Compatible API 的工具。

3. 多账号负载均衡

CLIProxyAPI 支持 Gemini、OpenAI、Claude 等多账号轮询，可以在多个账号之间分摊请求。

这对于额度有限、账号较多、需要稳定调用的场景比较有用。

4. 本地或服务器部署统一代理服务

CLIProxyAPI 可以部署在：

  • 本地电脑；
  • Linux 服务器；
  • Docker 容器；
  • 云服务器；
  • 局域网环境；
  • 反向代理后的 HTTPS 域名环境。

三、部署前准备

建议准备以下环境：

  • 一台本地电脑或服务器；
  • Git；
  • Go 环境，如果选择源码编译；
  • Docker，如果选择容器部署；
  • 一个或多个需要接入的账号，例如 Google、ChatGPT、Claude；
  • 一个自定义的 API Key，用于客户端访问 CLIProxyAPI。

CLIProxyAPI 默认监听端口通常为：

1

8317

如果只希望本机访问，可以让服务监听：

1

127.0.0.1

如果希望局域网或公网访问，可以让服务监听所有网络接口：

1

0.0.0.0

或在配置中将 host 留空。

四、安装方式一：macOS 使用 Homebrew

macOS 用户可以直接使用 Homebrew 安装：

1
2

brew services start cliproxyapi

使用 brew services 启动后，配置文件通常位于 Homebrew 的配置目录中。

Apple Silicon 设备一般是：

1

/opt/homebrew/etc/cliproxyapi.conf

Intel Mac 一般是：

1

/usr/local/etc/cliproxyapi.conf

如果希望继续使用用户目录下的配置文件，例如：

1

~/.cli-proxy-api/config.yaml

可以将 Homebrew 默认配置文件软链接到该位置：

1
2
3
4
5
6
7

brew services stop cliproxyapi

mv "$(brew --prefix)/etc/cliproxyapi.conf"    "$(brew --prefix)/etc/cliproxyapi.conf.bak"

ln -s ~/.cli-proxy-api/config.yaml    "$(brew --prefix)/etc/cliproxyapi.conf"

brew services start cliproxyapi

查看服务状态：

1

brew services list

停止服务：

1

brew services stop cliproxyapi

十三、OAuth 认证配置

CLIProxyAPI 的核心能力之一是通过 OAuth 登录接入 Gemini CLI、Codex、Claude Code 等服务。

1. 配置 Codex

本地环境中可以执行：

1

cli-proxy-api --codex-login

执行后会自动打开 ChatGPT 授权页面。

登录并授权成功后，认证文件会被保存到配置中的 auth-dir 目录。

如果你有多个 ChatGPT 账号，可以重复执行登录流程添加多个账号。

Docker 环境中可以执行：

1

docker compose exec cli-proxy-api /CLIProxyAPI/CLIProxyAPI -no-browser --codex-login

在服务器或容器环境中，通常没有浏览器，因此需要通过本地浏览器完成授权，并通过 SSH 隧道把 OAuth 回调转发回服务器。

2. 配置 Gemini CLI

Gemini CLI 认证通常需要 Google Cloud 项目 ID。

基本命令如下：

1

cli-proxy-api --login --project_id your-google-cloud-project-id

一般流程包括：

  1. 创建 Google Cloud 项目；
  2. 启用相关 Gemini API；
  3. 执行登录命令；
  4. 在浏览器中完成 Google 授权；
  5. 检查 auth-dir 中是否生成认证文件。

3. 配置 Claude Code

Claude Code 的 OAuth 认证方式与其他 OAuth 服务类似，通常也是执行对应的登录命令，然后在浏览器中完成授权。

如果你使用 Docker 或远程服务器，同样需要注意 OAuth 回调问题。

二十、使用 Nginx 反向代理

如果你希望通过域名访问 CLIProxyAPI，可以使用 Nginx 反向代理。

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://127.0.0.1:8317;
        proxy_http_version 1.1;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;

1

sudo certbot --nginx -d api.example.com

二十一、常见问题

1. 客户端返回 401 Unauthorized

通常是 API Key 不正确。

检查配置文件：

1
2

api-keys:
  - "sk-cli-proxy-demo-key"

客户端请求头应为：

1

Authorization: Bearer sk-cli-proxy-demo-key

2. 服务器部署后无法访问

检查容器是否运行：

1

docker ps

查看日志：

1

docker compose logs -f

确认端口是否监听：

1

ss -tulnp | grep 8317

检查防火墙：

1

ufw status

如果使用云服务器，还需要检查云厂商安全组是否放行 8317 端口。

3. OAuth 授权失败

远程服务器中 OAuth 失败，常见原因是回调地址问题。

Gemini CLI、Claude、Codex 的 OAuth 认证通常需要接收 localhost 回调。远程 Docker 环境需要 SSH 隧道或其他回调转发方式。

4. 多账号没有轮询

检查是否添加了多个认证文件，并确认配置中的路由策略。

示例：

1
2

routing:
  strategy: "round-robin"

常见策略包括：

策略 说明
round-robin 多个凭据之间轮询
fill-first 优先使用前面的凭据

5. 管理面板无法访问

检查配置：

1
2
3
4

remote-management:
  allow-remote: true
  secret-key: "your-management-key"
  disable-control-panel: false

如果 secret-key 为空，Management API 可能会被禁用。

6. 客户端请求没有响应或流式中断

可以检查：

  • 是否使用了正确的 /v1 路径；
  • Nginx 是否开启了缓冲；
  • Docker 容器日志是否有报错；
  • 目标账号是否额度耗尽；
  • OAuth 是否过期；
  • 模型名称是否正确；
  • 客户端是否兼容流式响应。

7. Docker 更新后认证丢失

通常是因为没有持久化 auth-dir。

Docker Compose 中必须挂载认证目录：

1
2

volumes:
  - ./auths:/root/.cli-proxy-api

否则容器重建后，OAuth 登录信息可能会丢失。

二十二、生产部署建议

如果要长期部署在服务器上，建议：

1. 使用 Docker Compose 管理服务

方便重启、升级、持久化配置。

2. 配置 HTTPS 反向代理

推荐使用：

  • Nginx；
  • Caddy；
  • Traefik。

3. 不要暴露无保护的管理面板

必须设置：

1
2

remote-management:
  secret-key: "strong-random-key"

4. API Key 使用高强度随机字符串

不要使用：

1
2
3

123456
test-key
sk-test

建议使用高强度随机字符串。

5. 认证文件目录必须持久化

Docker 部署时一定要挂载 auth-dir，否则容器删除后 OAuth 登录信息会丢失。

6. 定期查看官方 Releases 和 Issues

CLIProxyAPI 更新较快，建议定期查看：
